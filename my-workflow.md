# 我的工作习惯与规则

## 关于沟通方式
- 每次完成一个任务后，用产品经理能听懂的非技术语言向我解释你做了什么
- 遇到任何报错，先告诉我是什么问题、原因是什么，再问我是否动手修
- 如果有多种实现方案，列出来告诉我各自的优缺点，而不是直接选一个做
- 用简单直白的语言和我沟通，我是零代码基础的产品经理

## 关于开发节奏
- 每次只做我说的一件事，完成后停下来等我确认，不要自己继续做下一步
- 一次只改一件事，不要同时修改多个功能或多个问题
- 每完成一个完整功能，提醒我先在本地浏览器验收，验收通过后再做git commit
- 每次git commit，告诉我完整的commit命令让我直接复制执行，然后同步更新CLAUDE.md的当前进度
- 部署完成后，提醒我打开线上链接验收核心流程，不能只依赖本地验收

## 关于进度记录
- 每次git commit后，同步更新progress.md
- 当我说"结束""收工""明天继续"时，检查一次progress.md是否是最新状态
- progress.md里需要包含：已完成的功能、当前状态、下一步计划、未解决的问题

## 关于安全
- API Key和任何敏感信息只放在.env文件里，永远不写进代码
- 提醒我.env文件不能上传GitHub

## 关于报错处理
- 遇到报错不要自己反复猜测修改，告诉我完整的报错信息，等我确认再处理
- 如果同一个问题修了两轮还没解决，停下来重新分析原因，不要继续无效修改

## 关于新对话
- 每次新对话开始，主动提示我先读取CLAUDE.md和progress.md
- 读完后用一句话告诉我你理解的当前项目状态，等我确认再继续
- 如果是全新项目，在做任何事之前先帮我生成CLAUDE.md初稿，包含项目描述、技术栈、我的验收规则和偏好
