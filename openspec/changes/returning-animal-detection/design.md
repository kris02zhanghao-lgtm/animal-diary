## Context

当前产品每条偶遇记录是独立的，不同记录之间没有任何关联。用户在同一地点多次遇到同一只橘猫，在产品里是完全割裂的 3 条记录。v0.8 要在记录之间建立"可能是同一只"的关联关系，让产品具备跨时间的动物身份感知。

技术约束：
- 图片以 base64 存在 Supabase records 表，直接让 AI 对比两张图（multimodal）是可行路径
- 现有保存链路：前端 → `api/save-record.js` → Supabase，加回头客检测要在这里接入
- RLS 已开启，后端用 service role key，前端用 anon token

## Goals / Non-Goals

**Goals:**
- 保存新记录后自动检测同物种历史，返回最相似的历史记录 ID 和分数
- 相似度 ≥ 60 时弹出建议卡片，用户可确认或忽略
- 详情页展示该记录的回头客分数和关联入口
- 手动从详情页触发检测

**Non-Goals:**
- 不构建多条记录的"回头客网络"（v0.8 只建立 1:1 关联，不做 1:N）
- 不做图像 embedding / 向量搜索（复杂度过高，AI 对比足够）
- 不做 100% 准确的个体识别（定位为"可能是同一只"，不是科学级别）
- 不支持跨物种关联

## Decisions

### 决策 1：AI 对比方式 — 文字描述对比 vs. 多模态图像对比

**选择：多模态图像对比（传两张图给 AI）**

理由：
- 文字描述对比（让 AI 比较两段日志）准确度低，很多记录日志风格相似
- 多模态对比让 AI 直接看外观特征（毛色、花纹、体型、特征），更接近人眼判断
- Kimi-k2.5 / Gemini 都支持多图输入

被否定：文字描述 + 元数据对比
- 原因：识别准确度低，容易误判同物种不同个体

---

### 决策 2：物种分类多层结构

**选择：建立「大类 category」+ 「中类 species_tag」的两层分类**

- `category`（大类，6 个）：猫、狗、鸟、松鼠、兔子、其他 → 用于报告统计和图鉴第一层聚合
- `species_tag`（中类，约 93 个，见下文预定义列表）→ 用于回头客匹配、图鉴完成度、成就系统

理由：
- 原有的 `species` 字段是 AI 自由输出，不稳定（同一只猫识别为"橘猫"或"橘色条纹猫"）
- 引入 `species_tag` 作为标准化的中间层，既能提供足够的物种粒度，又能确保稳定性
- 保留 `category` 用于报告聚合（统计维度），保留 `species` 用于原始 AI 输出（日志展示）

---

### 决策 3：触发阈值 — 同中类历史记录数量限制

**选择：同中类（species_tag）历史记录 ≥ 1 条就触发**

理由：
- 用中类而非大类匹配，因为同大类的不同个体（黑猫和橘猫）不可能是同一只，按中类过滤更精准
- 每次保存多 1 次 AI 调用，成本可接受

拉取数量上限：最近 5 条（按时间倒序）
- 原因：5 条内 AI 对比结果更聚焦，时间跨度合理

---

### 决策 4：API 架构 — 同步 vs. 异步

**选择：保存后同步触发检测，在同一请求里返回检测结果**

理由：
- 简单可控，前端不需要轮询
- 当前用户规模下延迟可接受（多 1 次 AI 调用约 2-4s）
- 若同物种历史为 0 条，跳过检测，延迟为 0

被否定：异步触发（Vercel Background Function / 队列）
- 原因：架构复杂，前端需要轮询或 webhook，当前阶段不必要

---

### 决策 5：相似度存储方式

**选择：records 表新增 `species_tag` + `similar_record_id` + `similarity_score` + `confirmed_returning` 四列**

- `species_tag`：varchar，标准化中类物种（从预定义列表选择）；不可为 NULL
- `similar_record_id`：UUID，关联到另一条记录；NULL 表示未检测或无匹配
- `similarity_score`：integer 0-100；NULL 表示未检测；0 表示检测过但不相似
- `confirmed_returning`：boolean，default false；用户确认关联后改为 true

---

### 决策 6：相似度阈值

- 弹出提示：≥ 60 分
- 详情页展示（已检测，无论是否弹过）：≥ 40 分
- < 40 分：不展示，视为"不同个体"

### 决策 7：物种分类规则（species_tag 的归一化规则）

**选择：毛色优先于品种，除非品种特征极其明显**

- **猫**：优先用毛色标签（橘猫/黑猫/三花猫），仅当品种特征突出时用品种（英短/布偶/暹罗）
  - 示例：橘色英短 → 标记为"橘猫"；纯黑布偶 → 标记为"黑猫"；暹罗猫特征明显 → 标记为"暹罗猫"
- **狗**：优先用品种标签，品种不确定时选"中华田园犬"
  - 示例：明显的哈士奇特征 → 标记为"哈士奇"；混血看不出 → 标记为"中华田园犬"
- **其他**：按预定义列表直接映射，无二义

理由：
- 用户的日常认知就是这样的（路边看到的猫首先想到毛色，宠物狗首先想到品种）
- 毛色相比品种更容易被 AI 稳定识别

## Risks / Trade-offs

- **AI 对比成本**：每次保存多 1 次 AI 调用。若用户每天记录 10 条，1 个月约 300 次额外调用。当前成本可接受，但要在 Vercel 日志里监控
  → 缓解：对比时只取最近 5 条历史，不是全量

- **图片 base64 体积**：base64 传给 AI 体积大，响应慢
  → 缓解：历史记录的 base64 已存在数据库，不需要重新上传；可以在传给 AI 时截断到合理长度

- **AI 判断不稳定**：同一对图可能每次给出不同分数
  → 缓解：结果落库，只调用一次，不重复判断

- **用户不理解分数含义**
  → 缓解：UI 不暴露原始分数，只展示"相似度高 / 中 / 低"和具体描述

- **species_tag 映射出错**：AI 把识别结果映射到预定义列表时出错
  → 缓解：给 AI 一份清晰的列表，让它从中选择；兜底为"其他动物"

## Migration Plan

1. Supabase records 表添加迁移：`ALTER TABLE records ADD COLUMN species_tag VARCHAR NOT NULL DEFAULT 'other-animal', ADD COLUMN similar_record_id UUID REFERENCES records(id), ADD COLUMN similarity_score INTEGER, ADD COLUMN confirmed_returning BOOLEAN DEFAULT false`
2. 修改现有的 AI 识别流程（第 2 次 AI 调用）：改为同时返回 `category` 和 `species_tag`，按预定义列表和毛色优先规则映射
3. 部署新后端 `api/detect-returning.js`
4. 修改 `api/save-record.js` 接入检测，用 `species_tag` 而非 `species` 作为匹配维度
5. 部署前端新组件和详情页修改
6. 验证：新建一条与历史同物种（species_tag）记录，确认弹窗出现、分数正确写库

回滚：如果检测逻辑出问题，`save-record.js` 里用 try-catch 包住检测调用，失败时静默降级（只保存记录，不返回检测结果），不影响核心保存链路。

## Open Questions

- 第 2 次 AI 调用的完整 prompt：既要返回 `category` 大类，也要返回 `species_tag` 中类，需要明确告诉 AI 毛色优先规则和完整预定义列表
- `confirmed_returning` 字段的 RLS 规则：用户只能更新自己记录的该字段（已有 update 策略，应该满足）
- 历史数据的 `species_tag` 如何初始化：可以留作 v0.9 的优化，暂时新记录才有 species_tag
