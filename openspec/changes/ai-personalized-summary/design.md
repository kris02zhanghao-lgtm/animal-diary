## Context

用户在v0.8已经有了回头客识别功能，积累了丰富的历史记录数据。目前报告页（v0.6-v0.7）只展示静态数据汇总（总数、Top物种、Top地点、最活跃月份），缺乏"故事感"和个性化体验。

升级方向：从静态表格升级为**AI驱动的洞察生成**，让AI理解用户的观察模式，生成有趣且有个性的总结文案。

## Goals / Non-Goals

**Goals:**
- 在报告页新增个性化洞察section，展示AI生成的观察模式总结
- 支持「最近三个月」和「自然年」两个时间窗口下的洞察
- 数据分层：≥5条记录才生成洞察，<5条显示友好提示
- 洞察类型包括：时间偏好、地点特征、用户观察特征总结（如"你更容易在傍晚遇到鸟类"）
- 用户体验：加载态、成功展示、失败降级都有清晰反馈

**Non-Goals:**
- 不持久化存储洞察结果（每次打开都重新生成，保持数据新鲜）
- 不实现洞察缓存或预算管理（暂不考虑AI成本优化）
- 不支持洞察的分享或导出（后续版本考虑）

## Decisions

### 1. 洞察生成时机：实时生成 vs 预生成
**决策**：用户打开报告页时实时调用 `/api/generate-insights` 生成洞察（实时模式）

**考虑的方案**：
- 方案A（实时生成）✅ 选中
  - 优点：数据总是最新的、实现简单、无缓存复杂度
  - 缺点：每次打开报告都调用AI（成本），可能有延迟
- 方案B（定期预生成）
  - 优点：减少AI调用成本、加载快
  - 缺点：引入定时任务、缓存失效管理、数据可能过期

**理由**：报告页打开频率相对较低，用户更关心数据新鲜度而非加载速度。实现复杂度低，利于快速迭代。

### 2. 数据流向与隐私
**决策**：后端在生成洞察前，将用户的历史记录（species/location/timestamp等）聚合成结构化JSON发送给AI，但不包含用户身份信息。

**细节**：
- `/api/generate-insights` 接收 `timeWindow` 参数（"month" | "year"）
- 后端查询该窗口内的所有记录，按物种+时间+地点聚合
- 构造 Prompt 格式：
  ```
  用户的动物观察记录（时间段：最近三个月）：
  - 橘猫：5次，主要在公司附近傍晚遇到，也有晚上在家楼下遇到
  - 鸽子：3次，在公园中午遇到
  - 麻雀：2次，在办公室附近早晨遇到
  
  请生成3条有趣且准确的个性化观察洞察，每条不超过20字。
  ```
- AI返回结构化洞察数组

### 3. 洞察内容结构
**决策**：每个洞察是一句话（20字以内），包含：时间偏好 或 地点特征 或 观察行为特征

**示例**：
- "你是傍晚鸟类观察家" （观察行为特征）
- "橘猫最爱在你家楼下出没" （地点特征）
- "你在公司附近更容易遇到松鼠" （地点+物种组合）

**前端展示**：
- 洞察列表（3-5条）垂直排列
- 每条前面加emoji（🌙傍晚、📍地点、🐾动物等）
- 字体比数据更大，营造"故事感"

### 4. 错误处理与降级
**决策**：AI调用失败时，前端显示友好的降级文案而非报错

**处理逻辑**：
- 成功：展示AI生成的洞察列表
- 失败/超时：显示"数据整理中，下次打开会有更多发现"
- 数据不足（<5条）：显示"记录越多，洞察越有趣，继续探索吧！"

## Risks / Trade-offs

| 风险 | 影响 | 缓解方案 |
|------|------|--------|
| AI 生成质量不稳定 | 洞察可能不准确或重复 | Prompt工程：提供结构化数据、示例、字数限制 |
| API 响应慢（>3s） | 用户等待时间长 | 增加loading动画、可选地实现超时降级 |
| AI 成本上升 | 每条记录都生成洞察，API调用频繁 | 暂不考虑，v0.9可考虑缓存 |
| 用户期望与实际不符 | AI洞察可能与用户认知有偏差 | 所有洞察用"可能"、"往往"等修饰词，表示模式而非事实 |

**Trade-off**：简单实时生成 vs 复杂的缓存优化
- 选择前者，保持MVP简洁，后续如果成本成为瓶颈再优化

## Implementation Notes

- 后端会先对记录做 species/location/time 聚合，再把结构化统计发送给 AI，避免直接传原始长列表导致 prompt 漂移。
- 如果 AI 返回内容过短、重复或超出字数限制，服务端会过滤并用统计结果补足 3-5 条洞察，优先保证相关性和可读性。
- 前端 5 秒超时、后端 10 秒超时，任何一侧失败都降级为友好文案，不把错误细节暴露给用户。
