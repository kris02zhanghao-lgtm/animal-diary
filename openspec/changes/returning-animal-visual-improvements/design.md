## Context

当前回头客识别（v0.8）的前端交互和展示方式还需打磨。用户在确认弹窗中看不到被关联动物的照片，在详情页查看关联时信息单向且不完整，手动检测失败时没有反馈。这些问题分散在多个前端组件中，需要协调改进。

**现有实现：**
- `ReturningSuggestionModal.jsx`：展示弹窗，包含文案和分数，无图片
- `ListPage.jsx` 详情页：显示 `similarity_score` 和 `similar_record_id`，可跳转到关联记录
- 详情页菜单：「查找回头客」选项，点击调用 `detectReturning()` API，成功或失败无反馈

## Goals / Non-Goals

**Goals:**
- 弹窗中展示被关联动物的照片，帮助用户直观确认
- 双向展示关联关系：A 关联到 B，则 A 和 B 都能看到彼此的关联
- 手动检测失败时给用户反馈提示（如"未找到相似记录"）
- 提高"查找回头客"功能的可发现性，从菜单移到详情页顶部

**Non-Goals:**
- 不修改后端检测算法或分数逻辑
- 不支持编辑或删除已建立的关联（保持当前行为）
- 不实现全局的关联关系图可视化

## Decisions

### 决策 1：弹窗中的图片获取方式
**方案：** 在 `saveRecord` 返回的 `returningDetection` 中包含 `similarRecord` 对象，其中包含 `image_base64`。弹窗直接从 props 读取，无需额外查询。

**理由：** 
- 减少 API 调用，利用已有的返回数据
- 弹窗渲染时图片已经可用，不需要 loading
- `similarRecord` 在保存时已经从 `detectReturning.js` 获取，成本已付出

**备选方案考虑：**
- 弹窗显示时再查一次后端：会增加延迟和 API 调用，不必要

### 决策 2：双向关联展示策略
**方案：** 在 `list-records.js` 的返回数据中新增 `linked_from` 字段，包含所有指向该记录的关联（通过 `similar_record_id` 反向查询）。详情页同时显示 `similar_record_id`（该记录指向谁）和 `linked_from`（谁指向该记录）。

**理由：**
- 保持数据库表结构不变，只修改查询逻辑
- 一次详情页加载就能获得完整关联信息
- 详情页 UI 可以分两个区块展示："我指向的关联" 和 "指向我的关联"

**备选方案考虑：**
- 数据库表新增反向引用列：会增加迁移复杂度，也违反规范化，不必要

### 决策 3：检测反馈展示位置
**方案：** 手动检测时，使用现有的 `setSaveError()` state（或新增 `setDetectionStatus` state）在详情页顶部或菜单旁边显示短暂提示。
- 检测中：菜单项旁边显示 loading 或按钮变灰
- 检测完成找到结果：弹出确认弹窗（现有逻辑）
- 检测完成找不到：页面顶部显示 toast 或内联文案"未找到相似记录"

**理由：** 复用现有状态管理和 UI 模式，保持一致性。不需要新增复杂的模态框。

### 决策 4："查找回头客"按钮位置
**方案：** 在详情页菜单区域，于「编辑」按钮右下方新增一个 icon 按钮（或文字按钮），标签"🔍 查找回头客"。点击触发同样的检测逻辑。菜单中的同名选项保留（后续可考虑移除）。

**理由：** 
- 提高易发现性，不隐在菜单里
- 让功能和「编辑」并列，强调这是重要交互
- 逐步迁移：先保留菜单选项，新增按钮，用户习惯后再下线菜单项

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| `list-records.js` 反向查询可能影响性能（查询所有指向该记录的记录） | 在 `linked_from` 结果上限制条数（如最多返回 3 个），或只在详情页展开时才加载反向数据 |
| 双向关联在 UI 中可能造成信息过载 | 只显示最相似的 1 条反向关联，不显示所有关联 |
| 手动检测反馈位置不够醒目 | 可考虑用小型 toast 通知而非页面内提示 |

## Migration Plan

1. **第一步**：修改 `ReturningSuggestionModal.jsx`，接收 `similarRecord` props，展示照片
2. **第二步**：修改 `api/list-records.js`，查询 `linked_from` 数据，返回反向关联
3. **第三步**：修改 `ListPage.jsx` 详情页，展示反向关联信息、添加"查找回头客"按钮、添加检测反馈
4. **第四步**：测试并上线，观察反向查询性能

## Open Questions

1. 反向查询的数据量：平均有多少条记录的 `similar_record_id` 指向同一条记录？需要限制返回条数吗？
2. 手动检测反馈的持续时间：成功/失败提示应该显示多久？用户需要主动关闭还是自动消失？
3. "查找回头客"按钮的具体样式和文案：icon + 文字还是纯 icon？放在菜单按钮左边还是右边？
