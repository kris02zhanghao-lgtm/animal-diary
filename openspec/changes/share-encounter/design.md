## Context

当前 DetailPage 顶部有菜单按钮（⋮），菜单选项包括「分享」「删除」。「分享」选项目前显示"即将上线"。

ReportPage 新增后，用户可以在报告中看到 top 物种卡片，这些卡片也应该支持分享。

## Goals / Non-Goals

**Goals:**
- 让用户能一键生成分享卡片
- 分享卡片包含足够的信息：照片、物种名、地点、日期、日志摘要
- 支持多种分享方式：图片下载、复制文案、一键复制链接（Clipboard API）
- 分享卡片样式精致，符合星露谷像素风格

**Non-Goals:**
- 不支持直接分享到社交平台（用户手动粘贴）
- 不生成二维码
- 不支持自定义分享卡片样式

## Decisions

### D1：分享卡片格式
**选择：** Canvas 生成动态图片（PNG），同时保留文案备选

**理由：**
- Canvas 生成的图片在所有平台通用，兼容性最强
- 图片版本视觉冲击力大，适合朋友圈/小红书
- 文案版本是降级方案，防止 Canvas 兼容性问题

**替代方案：**
- ❌ HTML 截图（html2canvas）：需额外依赖，性能差
- ❌ 纯文案分享：低转化，不符合"漂亮卡片"的目标

### D2：卡片内容和布局
**选择：** 纵向卡片（600x800px），分为三区域：
1. 照片区（上方，object-fit cover）
2. 信息区（中间，物种名、地点、日期）
3. 日志区（下方，截断前 100 字）

**理由：**
- 600x800 适合手机竖屏截图和分享
- 三区分层清晰，信息层级明确
- 与现有卡片设计风格统一

### D3：分享入口
**选择：** DetailPage 菜单 + ReportPage 物种卡片快捷分享

DetailPage 菜单：
- 现有的「分享」选项：生成当前偶遇的分享卡片

ReportPage 物种卡片（可选）：
- Top 3 物种每个卡片右下角加「分享」icon
- 点击生成该物种的汇总卡片（如"2026年你遇见了 5 只猫"）

**理由：**
- DetailPage 是主要的分享入口（用户最常用）
- ReportPage 的分享是增强体验（非必需）

### D4：跨浏览器兼容性降级
**选择：** Canvas 生成失败时，弹窗显示纯文案版本，用户手动复制

```
【偶遇记录分享】
🐱 橘猫
📍 北京，798艺术区
📅 2026-04-27
日志：你在这个闲适的下午...（前100字）
```

**理由：**
- Canvas 在 99% 的现代浏览器上都支持
- 降级方案确保老旧设备也能分享
- 简化处理，不需复杂的错误恢复逻辑

## Risks / Trade-offs

| 风险 | 缓解方案 |
|------|---------|
| Canvas 绘制图片时 Base64 照片数据大，可能导致卡顿 | 使用 requestAnimationFrame，加载中显示 spinner |
| 分享卡片在不同手机上显示效果不一致 | 卡片样式固定尺寸，使用系统字体确保兼容性 |
| 用户点击分享后期待自动分享到社交平台 | 初期只支持复制/下载，文案中注明"自行粘贴到朋友圈" |

## Migration Plan

1. 新建 `src/utils/shareUtils.js`：Canvas 生成卡片的核心逻辑
2. 修改 `src/pages/DetailPage.jsx`：菜单「分享」选项变为实际功能
3. 修改 `src/pages/ReportPage.jsx`（可选）：物种卡片新增分享按钮
4. 新建共享弹窗组件（如需要）
5. 本地测试：生成图片 → 验证样式 → 验证下载/复制功能
6. 线上验证：在真实设备上测试分享流程

## Open Questions

- Q：ReportPage 中的分享卡片应该显示什么？（单条记录 vs 该物种的汇总）
  A：建议 v0.7 先只做 DetailPage 的单条分享，ReportPage 的物种卡片分享留给后续迭代

- Q：图片格式是否需要支持 JPEG 以减小文件大小？
  A：先用 PNG 确保质量，后续优化

- Q：分享时是否需要添加"分享自 xxx 产品"的 watermark？
  A：暂不添加，保持简洁；产品推广通过内容本身的分享激励
