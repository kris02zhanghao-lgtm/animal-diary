## Context

当前 ListPage 只实现了时间线视图（按季度分组的卡片轮播）。首页底部 tab 栏已有「图鉴」tab，但点击时显示空白或占位。现在需要实现图鉴视图，将所有偶遇记录按物种聚合，展示用户的"收集成果"。

数据来源：records 表，通过 getRecords 获取。每条记录包含 species、location、createdAt、imageBase64 等字段。

## Goals / Non-Goals

**Goals:**
- 实现图鉴视图，按物种分类展示用户遇到过的所有物种
- 每个物种卡片显示：代表性照片（最新一次）、名称、遇到次数、最近地点
- 点击物种卡片可展开该物种的所有记录时间线
- 计算并展示物种收集完成度

**Non-Goals:**
- 不实现成就系统（v0.6）
- 不做分享卡片功能（v1.0）
- 不做回头客识别（v0.6）
- 不做物种搜索或筛选（暂不需要）

## Decisions

**决策 1：数据聚合位置（前端 vs 后端）**
- 选择：前端完成聚合
- 理由：记录数量目前不大（几十条），前端基于 records 数组做 group by + reduce 足够快。后端做的话需要新增 API，这个版本优先简化。
- 备选方案：后端新增 `/api/collection-stats` 接口，Supabase 直接返回聚合数据。如果将来记录数超过 1000+ 再迁移。

**决策 2：物种卡片布局**
- 选择：网格布局（3 列或 2 列响应式）
- 理由：更符合"图鉴"的视觉感受，让用户一眼看到全景。列表布局太线性。
- CSS：Tailwind grid-cols-2 lg:grid-cols-3，gap-4

**决策 3：物种"代表性照片"选择**
- 选择：按最新一次的照片
- 理由：简单直接，用户最近才拍的。无需额外逻辑。
- 备选方案：选质量最好的（需要主观评分），或者所有照片轮播。暂不做。

**决策 4：每个物种卡片显示百分比**
- 选择：显示"该物种遇到次数 / 总记录数 = 百分比"，而不是物种数百分比
- 理由：物种数百分比没有明确的"理论最大值"（城市有多少种动物？不好定）。但每个物种的"出现频率"百分比有意义，且为成就系统铺路（比如"集齐 10 次猫"）。
- 实装：卡片显示"遇到3次 (15%)"，15% 代表 3/(总记录数 20) = 15%
- 备选方案：显示"3/10"（集齐 10 次解锁成就），但这需要预定义每个物种的目标值，v0.5 暂不做。

**决策 5：图鉴作为独立 tab 页面**
- 选择：图鉴从底部 tab 栏直接进入，不在 ListPage 内部做 tab 切换
- 理由：底部 tab 栏已有"时间线"、"地图"、"图鉴"等选项，用户可直接切换。在 ListPage 内部再加小 tab 会造成交互冗余。
- 技术：在 App.jsx 的路由中，点击底部"图鉴"tab 显示 CollectionPage（或在 ListPage 中独立组件）

## Risks / Trade-offs

- [聚合性能] 如果记录数超过 1000+，前端聚合会变慢 → 后续迁移到后端 API
- [百分比不明确] 现在不显示完成度百分比，可能降低"收集欲" → 后续根据用户反馈决定是否补充
- [物种去重] 假设 species 字段完全匹配才认为是同一物种（如"橘猫"和"橙色虎纹猫"被当作两种）→ 暂时接受，后续可加 AI 物种标准化或用户手动合并
- [更新延迟] 新增记录后，图鉴页需要手动刷新或切换 tab 才能看到 → 可在保存成功后主动调用 fetchRecords 更新 state

## Migration Plan

1. 新建 CollectionPage.jsx，作为独立的图鉴页面组件
2. 第一层：物种卡片网格展示，调用 getSpeciesStats() 获取聚合数据，卡片显示代表照片、物种名、遇到次数、百分比
3. 第二层：点击物种卡片后展开该物种的照片网格，点击照片查看完整卡片详情
4. 新建 services/collectionService.js，提供 `getSpeciesStats(records)` 聚合函数，返回 `[{ species, count, percentage, latestRecord, ... }]`
5. 确保从其他页面返回图鉴时自动刷新数据（调用 fetchRecords）
