## Context

当前NewEncounterPage的流程：
1. 用户上传照片 → 状态显示"上传中"或图片预览
2. 点击"AI帮我生成" → `recognizeAnimal()` 调用后端 `/api/recognize`
3. 成功 → 显示卡片，包含AI返回的species/journal
4. 失败 → 显示错误信息，用户卡住，无法继续

现有卡片状态：只支持"识别成功"的展示，物种和日志来自AI。

后端`api/recognize.js`当识别失败时返回：
```json
{ error: "Recognition failed" }
```
前端收到错误后直接展示提示文本，不尝试显示卡片。

## Goals / Non-Goals

**Goals:**
- 识别失败/超时时仍显示卡片框架（照片 + 空物种 + 空日志），允许用户手动填充
- 保存按钮在识别失败场景也可用，支持保存用户手填的记录
- 区分识别成功vs失败两种卡片状态（视觉上需要有差异提示）
- 识别失败的错误信息改为卡片下方提示，不遮挡卡片操作

**Non-Goals:**
- 不涉及识别重试机制（网络恢复后自动重新识别）
- 不改变识别超时时间配置（保持现有25s超时）
- 不支持部分识别成功（比如识别出物种但日志生成失败时自动填物种）——暂不处理，所有失败都作为完全失败处理
- 不修改后端API合约（仍返回error）

## Decisions

1. **卡片状态管理**
   - 新增`recognitionFailed` boolean状态，为true时表示识别失败但显示空卡片
   - 当`recognitionFailed=true`时，卡片内species和journal字段为用户编辑的新值或空值
   - 不新增API字段，完全在前端状态层处理

2. **保存逻辑**
   - 识别失败时不再禁用保存按钮，允许空物种保存（后端有字段非空约束则前端需要校验）
   - 实际上按当前的卡片设计，species为空时保存按钮已禁用，这里维持现状
   - 但允许**至少填一个字段（物种或地点或日志）**就可保存，物种优先校验

3. **错误提示位置**
   - 识别错误改为显示在卡片底部（AI按钮下方）的红色/警示色提示
   - 格式："❌ AI识别失败，请手动补充"
   - 相比目前的大提示框，这样既通知用户又不阻挡操作

4. **视觉区分**
   - 识别失败时，卡片背景不变，但在顶部或物种区加个"⚠️ 手动填充"的轻量标签
   - 或直接在species输入框placeholder改为"AI识别失败，请输入物种"

5. **改动最小化**
   - 只改NewEncounterPage.jsx的状态逻辑和条件渲染
   - EncounterResultCard组件新增`isFailed`prop，条件渲染失败状态提示
   - 不改api/recognize.js（保持现有error处理）

## Risks / Trade-offs

- **[风险] 用户不知道何时手填**：识别失败时用户可能以为是卡片加载中，误点保存空白卡片
  - **缓解**：加明显的"手动填充"提示文本，让失败状态一眼看清
  
- **[风险] 前端只校验species非空，仍可能保存信息不完整的记录**：比如物种填了但日志/地点都空
  - **缓解**：这是现有设计允许的（用户可以保存不带日志的记录），不新增校验规则

- **[权衡] 识别失败后能保存，可能导致数据库里有大量用户手填的、质量参差的记录**
  - **接受**：这是feature的代价，好过用户因网络问题丢失照片。长期可加审核或手填字段标记
