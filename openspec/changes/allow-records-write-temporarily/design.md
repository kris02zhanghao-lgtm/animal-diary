## Context

当前线上流程已恢复 AI 识别，但在“保存到日志”阶段被 Supabase `records` 表的 RLS 策略拒绝。前端尚未接入匿名登录，因此任何依赖 `auth.uid()` 的写入策略都会拦截当前请求。这个变更的目标是用最小范围的数据库策略调整，先恢复记录写入能力。

## Goals / Non-Goals

**Goals:**
- 在不引入匿名登录的前提下，恢复前端对 `records` 表的新增写入。
- 保持现有前端代码和部署方式基本不变，减少继续返工。
- 将临时方案边界写清楚，便于后续切换到正式 RLS。

**Non-Goals:**
- 不实现用户身份体系或匿名登录。
- 不解决多用户数据隔离问题。
- 不调整 AI 识别、列表展示或 Supabase 表结构。

## Decisions

- 通过调整 Supabase `records` 表的 INSERT 策略，允许 `anon` 角色临时写入。
  - 原因：当前故障点在数据库权限，直接改策略是恢复保存功能的最短路径。
  - 备选方案：立即补匿名登录。该方案更完整，但涉及前端初始化、会话管理、RLS 联动，超出“先恢复可用”的范围。
- 保持前端仍使用现有 `supabaseService.js` 写入，不新增后端代理保存接口。
  - 原因：当前保存请求已经成功到达 Supabase，只是被权限拒绝，改前端调用路径不能解决根因。
- 将该方案定义为临时恢复方案，后续需要以匿名登录 + 用户隔离策略替换。
  - 原因：公开写入虽然能恢复可用，但不适合作为长期正式上线策略。

## Risks / Trade-offs

- [所有用户可写入同一张表] → 后续尽快补匿名登录和用户隔离，并在产品测试阶段控制使用范围。
- [公开写入可能引入脏数据] → 仅保留当前必要字段写入，不在本次临时方案中扩大表能力。
- [临时策略遗留过久] → 在 progress.md 和 proposal 中明确标记为过渡方案，避免被误认为最终方案。

## Migration Plan

1. 在 Supabase 为 `records` 表新增或调整临时 INSERT 策略，允许 `anon` 角色写入。
2. 保持现有前端代码不变，验证线上“保存到日志”恢复可用。
3. 记录该策略为短期方案，后续单独推进匿名登录 + 正式 RLS 变更。
4. 若需要回滚，删除该临时策略并恢复原有严格 RLS。

## Open Questions

- 是否同时放开 SELECT，还是仅恢复 INSERT 以满足当前最小目标？
- 正式方案是否采用 Supabase 匿名登录，还是改为后端代理写入？
