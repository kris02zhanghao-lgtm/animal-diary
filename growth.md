## 2026-04-27 · v0.4 收尾 + v0.4.1/v0.4.2 规划

### 发生了什么

修复了两个 bug（后端 AI 识别超时 + 错误提示位置），然后规划了 v0.4.1（位置选择器）和 v0.4.2（详情页编辑记录）。

### 关键决策

**为什么 v0.4.1 从「一个按钮」变成了「一个完整组件」**

原计划 v0.4.1 只是在详情页加一个「补充定位」按钮，触发 GPS。但仔细想之后发现：
- 地点字段有两个部分：文字地名 + GPS 坐标，只改一个会导致数据不一致
- 用户不总是实时补录，事后想补地点时 GPS 可能不准，需要搜索框
- v0.4.2 的「编辑记录」里地点字段也面临同样问题

所以把两个版本共用的底层能力（LocationPicker 组件 + update-record 接口）提前做好，避免重复造轮子。

### 学到的方法

**改「已有数据」比加「新功能」复杂**

原因：新功能出错了重来就行；改已有数据，改完之后整个系统的数据要还是一致的——列表、详情、地图都要对得上。估工作量时要乘以 1.5-2。

**四个规划问题**

规划任何一个功能时，先问自己：
1. 这个功能改的是哪个数据字段？这个字段在哪些地方被用到？
2. 用户有几种路径会用到这个功能？（不只是顺利路径，还有出错、没网、事后补录）
3. 这个功能和已有功能有没有可复用的底层能力？
4. 这个版本做完，下一个版本会不会因为它更容易或更难？

**先想「坏的情况」，再想「好的情况」**

大多数人规划时只跑顺利路径。但真实世界里，顺利路径只占一半，另一半是没权限、没网、数据缺字段、用户中途退出……把「坏的情况」想清楚了，工作量和复杂度才是真实的。

### 对比赛答辩的启发

评委想看的不只是产品 demo，而是：
- 你为什么选这个需求做
- 你规划功能时发现了什么坑、怎么提前绕开的
- 你怎么在有限时间里判断哪些做、哪些不做
- 你踩过什么错，复盘出了什么结论

**业务和代码可以学，但做事的态度和方法要跑在前面。**

## 2026-04-27 · v0.4.1 LocationPicker 实现

### 发生了什么

完成了 v0.4.1：LocationPicker 组件支持 GPS 自动定位 + 逆地理编码 + POI 搜索，详情页「修改定位」功能上线。

### 关键决策

**为什么 LocationPicker 做成底部弹出面板 + 双 tab**

- 「自动定位」解决「我就在这里」的场景
- 「搜索地点」解决「事后补录」的场景，GPS 不准确或用户只想记大概位置时非常有用
- 底部面板不遮挡主内容，手机上操作更自然

**高德 API 调用放在前端还是后端？**

选前端：高德 Key 本来就是通过域名白名单暴露的，官方推荐用法。如果后端 proxy 反而多一层转发，增加延迟和复杂度。

**update-record 接口设计成 partial update**

v0.4.1 更新 location 三字段，v0.4.2 更新 title/species/journal，接口复用。设计时就考虑后续版本，避免重复建接口。

### 踩的坑

**组件化思维不够早**

一开始只在 ListPage 里想怎么改地点，没意识到新建偶遇页以后也需要这个能力。好在 v0.4.1 规划时识别出来了，做成了可复用组件。

### 学到的方法

**边界情况要产品化，不能只是「报错」**

- GPS 权限被拒绝 → 不是弹出 technical error，而是显示「请在系统设置中开启」
- 逆地理编码失败 → 兜底显示「已定位」+ 坐标，不阻塞用户保存
- 搜索结果为空 → 显示「未找到相关地点，请尝试其他关键词」

好的错误提示是产品体验的一部分。

### 技术债记录

- 新建偶遇页的自动定位还没升级成 LocationPicker（用了旧逻辑），等 v0.4.2 完成后统一升级
- POI 搜索目前只返回 5 条，如果搜索结果很多，用户无法翻页（当前用户量下够用）

## 2026-04-27 · v0.5.1 长按多选删除 + 点击跳详情

### 发生了什么

图鉴第二层的"长按多选批量删除"和"点击跳转详情页"看似简单的两个交互，前后改了三轮才落地。中间踩了三个坑，每个都很有教育意义。

### 踩的坑

**坑 1：build 通过 ≠ 功能正常**

第一次实现时，代码里写了 `React.useRef(null)`，但 import 语句只有 `useState, useEffect` —— 根本没引入 React。`npm run build` 居然通过了，但运行时组件直接崩溃，导致**第二层所有事件监听器都没绑定上**——用户点击、长按、右键，全都没反应。

之前我判断"build 成功就算完成"，跳过了浏览器实际验证这一步，错过了这个明显的 ReferenceError。

**坑 2：在 map() 里调用 hooks 是反模式**

```jsx
{records.map((record) => {
  const ref = useRef(null)  // ❌ 违反 Rules of Hooks
  // ...
})}
```

即使 React 正确 import，这种写法每次渲染都会重新创建 ref，导致状态丢失（长按定时器永远不会真正生效）。**正确做法是把照片卡片提取成独立的子组件 `PhotoCard`，hooks 在顶层调用。**

**坑 3：用户说"完全没改变" → 我假设是代码问题，重构了一轮**

代码 + 部署 + 缓存，是三个独立变量。用户报"线上完全没反应"时，我没有先问"你在哪里测的"，而是默认是代码逻辑错了，又重构了一遍。结果发现：**代码改了但根本没 push 到 Vercel**——线上看到的是旧版本。三轮往返多花了大半小时。

### 关键决策

**用 OpenSpec 流程结构化变更**

这次用 `/opsx:propose` + `/opsx:apply` 把改动拆成 proposal → design → spec → tasks 四层，好处是：
- 决策点（如"为什么用定时器区分点击和长按"）被显式记录在 design.md
- 任务列表是可勾选的，避免漏掉边界情况（如 onTouchMove 取消长按）
- 后续归档可以追溯"当时是怎么决策的"

代价是前期慢了一点（5 分钟规划），但避免了边写边改的反复。

### 学到的方法

**Bug 报告排查的第一性原则：先问"在哪"，再问"为什么"**

用户报"不工作"时，按这个顺序追问：
1. **环境**：本地 / 线上 / 移动端？（缓存、部署、分支三大坑都在这一层）
2. **现象**：具体看到什么？（截图、控制台报错）
3. **再现**：什么操作触发的？

之前我经常跳过第 1 步直接进第 2、3 步，假设"代码就是问题"。这次教训深刻。

**「build 通过」不是验收标准，「肉眼看到功能跑起来」才是**

CLAUDE.md 里早就写了"对 UI 或前端改动，启动 dev server 在浏览器实际操作过"。这次跳过了这一步，导致一个明显的 ReferenceError 漏到用户那里。**任何前端改动，编译通过只是起点，浏览器实际跑一遍才算完成。**

**React Hooks 在循环里调用是常见反模式，识别信号：**

看到 `.map(item => { useXxx(...) })` 就要警觉——这几乎一定是错的。正确做法是把循环里的子节点提取成独立组件，让 hooks 在新组件的顶层调用。

### 对比赛答辩的启发

评委如果问"你怎么定位 bug"，可以讲这个故事：
- 第一次失败：build 通过但运行时崩溃 → 教训是"编译 ≠ 运行"
- 第二次失败：诊断方向错了，假设是代码问题，结果是部署问题 → 教训是"先验证环境再验证代码"

这种"踩坑 → 复盘 → 改进诊断流程"的故事，比单纯展示"功能做出来了"有说服力得多。

## 2026-04-27 · v0.5 + v0.5.1 产品视角复盘

### 为什么单独记一份产品视角

之前的复盘几乎都是技术视角（React Hooks、build 通过 ≠ 运行通过、bug 排查方法论）。但回头看本次会话，**真正决定产品质量的不是踩了哪些技术坑，而是做了哪些产品决策**——这些决策当时被夹在"bug 修复"和"功能实现"里，没被识别为产品决策。

这份复盘的目的是把这些决策**显式拎出来**，建立产品直觉。

### 这次会话实际上做了 3 个真正的产品决策（但当时没意识到）

**决策 1：图鉴的"1 张卡片 = 1 个大类"，不是 1 个品种**

- 当时是作为 bug 修复讨论的（"按 species 聚合会把同一只猫拆成多张卡"）
- 真正本质：**用户打开图鉴想看到的是"我遇到过几种动物"，不是"我有多少种猫的细分品种"**
- 按品种聚合：重度用户图鉴变成"橘猫 1 张 + 虎斑猫 1 张 + 英短 1 张……"，成就感被碎片化稀释
- 按大类聚合：「猫（遇到 12 次）」才有"在城市收集动物"的游戏感

**通用规律：** 任何聚合页（图鉴/统计/排行榜），先问两个问题再决定数据维度——
1. 用户视角的 1 个单元代表什么？
2. 重度用户用 1 年后这一页会变成什么样？（这个问题能预测聚合维度选错的灾难）

---

**决策 2：长按多选 = 借用系统范式，不发明新交互**

- kris 原话："类似相册那种长按编辑删除"
- 这句话背后是**借用用户已有的心智模型**（iOS Photos / 微信图片选择都用这个）
- 如果换成"加一个批量管理按钮，进入管理模式"——多了学习成本，体验差一截

**通用规律：** 设计交互前先问"系统/同类产品里这个动作的'母语'是什么"。有现成母语就直接用，发明新交互的门槛要极高。

---

**决策 3：用产品架构兜底 AI 的不稳定性**

- AI 识别同一只橘猫，可能这次说"橘猫"、下次说"虎斑猫，橘色条纹毛"——这是 AI 产品的本质问题
- 解法不是"训练 AI 更准"（不可控），而是**两次 AI 调用做归一化**：
  - 第 1 次：识别具体品种（信息丰富，给详情页用）
  - 第 2 次：映射到稳定大类（聚合稳定，给图鉴用）
- 这是 **AI 产品的通用模式**：用「AI 自由发挥层 + 产品稳定收敛层」两层夹住 AI 的不稳定

**通用规律：** 做 AI 功能时主动用这个夹层模式。不要试图让 AI 完美一致，要在产品层吸收 AI 的偏差。

### 产品价值密度的发现：小改动 ≠ 低价值

| 功能 | 用户感知 | 工程成本 | 价值密度 |
|------|---------|---------|---------|
| v0.5 图鉴第一层网格 | ⭐⭐⭐⭐ | 中 | 高 |
| v0.5 物种大类分类 | ⭐⭐ | 中（多一次 AI 调用） | 中 |
| v0.5.1 长按多选删除 | ⭐⭐ | 中 | 中 |
| **v0.5.1 点击跳转详情** | **⭐⭐⭐⭐** | **低（半小时）** | **极高** |

**洞察：点击跳转详情这件事工程成本最低、用户感知价值最高**——它把图鉴从"展示"变成"可探索"，补全了"图鉴 ↔ 时间线 ↔ 地图"的链接。但因为它"看起来太简单"，最初没作为独立功能规划，被埋在 v0.5.1 里和长按多选一起做。

**通用规律：** 不要被"小改动"骗了。**很多极高价值的功能恰恰是改动小的——因为它们补全了产品的链接，而不是新增内容**。规划版本时专门列一栏"打通现有功能的小改动"，优先级高于"加新功能"。

### 暴露的产品流程问题

**问题 1：验收没产品化**

每次反馈用的是用户语言（"图鉴里只有一种叫其他"、"完全没改变"、"长按没反应"）。这些反馈很珍贵，但处理是 ad-hoc 的，没沉淀成结构化 checklist。

**改进：** 给图鉴这种核心页面建一个 **30 秒可执行的冒烟 checklist**，每次部署后自己跑一遍。这 5 个 yes 比"代码改完了"更重要：
1. 图鉴第一层有几个大类？数据对不对？
2. 点击大类能进第二层吗？
3. 第二层照片能点开详情吗？
4. 长按/右键能进选择模式吗？
5. 删除一张照片能正确减少计数吗？

---

**问题 2：版本节奏被"急于推进"绑架**

时间线：19:04 v0.5 基础 → 19:46 直接验收（没充分自测）→ 验收漏掉归"其他"问题 → 返工 → 20:19 v0.5.1 又出 React 崩溃 + 没 push → 加倍返工。

模式：**急于推进版本号 → 验收草率 → 用户测试时才发现问题 → 加倍时间返工。**

**改进：** 每个版本上线后强制 **5 分钟"用户体验时间"**，期间不写代码、不和 AI 对话，纯粹当用户用一遍。这 5 分钟省下的返工时间是 30 分钟起步。

---

**问题 3：和 AI 协作时反馈语言之间需要"翻译层"**

"长按没反应"作为反馈，AI 默认假设是代码问题——但真相可能是部署/缓存/分支。三轮往返耗时 45 分钟。

**改进：** 反馈"功能 X 不工作"时，主动加三句话——
- 在哪里测的（线上 / 本地 / 手机）
- 看到了什么（截图、控制台报错）
- 做了什么操作

这三句话能让 AI 排查效率翻倍。

### v0.6 之前必须想清楚的产品策略问题

**问题 1：产品定位是「工具」还是「游戏」？**

当前的图鉴 + emoji + 像素风强烈指向"游戏化收集"，但核心交互（拍照 + AI 识别 + 保存）又像"日记工具"。两种定位的演进路径完全不同：
- **工具路径**：v0.6 做导出、统计、搜索、跨设备同步
- **游戏路径**：v0.6 做成就系统、打卡日历、稀有度系统、好友炫耀

这个答案决定 v0.6/v0.7/v1.0 的所有功能选择。

---

**问题 2：「记录」和「分享」哪个是核心？**

- 时间线 + 图鉴 + 地图 = 私人查看
- 但分享按钮一直是"敬请期待"

如果分享是核心，整个产品应围绕"生成一张可炫耀的偶遇卡"设计——AI 文案要有梗、图鉴成就感要强、分享链接要好看。

如果分享只是辅助，明确告诉自己"这是私人记录工具"，停止把"分享"当占位。

---

**问题 3：用户量增长前最先崩的是什么？**

技术债清单里记的是"图片 base64 存数据库"。但产品视角看更早崩的可能是：
- **新用户引导路径**：打开 app 是空白页，不知道第一步该干啥
- **AI 识别失败的产品体验**：现在是错误提示，应该是"AI 失败也能保存（手填即可）"——其实已做但没被强调
- **国内访问困难**：vercel.app 域名 = 没有真正可触达用户

这三个里，技术含量最低的（域名）反而对产品可触达性影响最大——它需要钱和决心，不需要写代码。

### 学到的方法

1. **聚合页两问法**：1 个单元代表什么？重度用户 1 年后这页变成什么样？
2. **交互"母语"原则**：先找系统/同类产品的现成范式，再考虑发明新交互
3. **AI 夹层模式**：自由发挥层 + 稳定收敛层，吸收 AI 的偏差
4. **小改动可能极高价值**：版本规划专列"打通现有功能的小改动"
5. **5 分钟用户体验时间**：上线后强制冷却，纯当用户用一遍
6. **反馈三句话**：在哪 / 看到啥 / 做了啥，让 AI 排查效率翻倍

### 对比赛答辩的启发

可以讲一个"被埋没的产品决策"的故事——
- 表面：v0.5 上线后发现物种全归"其他"，改了 4 次 prompt 才解决
- 真相：这是个**聚合维度选错**的产品判断问题，不是 AI 问题
- 反思：作为产品经理，**我开始有意识地把"产品决策"和"技术问题"区分开**——之前所有问题在我脑子里都是混在一起的"bug"

这种"从技术视角升级到产品视角"的故事，比展示具体功能更有说服力——评委想看的是**产品经理的成长**，不是产品本身。
