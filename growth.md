## 2026-04-27 · v0.4 收尾 + v0.4.1/v0.4.2 规划

### 发生了什么

修复了两个 bug（后端 AI 识别超时 + 错误提示位置），然后规划了 v0.4.1（位置选择器）和 v0.4.2（详情页编辑记录）。

### 关键决策

**为什么 v0.4.1 从「一个按钮」变成了「一个完整组件」**

原计划 v0.4.1 只是在详情页加一个「补充定位」按钮，触发 GPS。但仔细想之后发现：
- 地点字段有两个部分：文字地名 + GPS 坐标，只改一个会导致数据不一致
- 用户不总是实时补录，事后想补地点时 GPS 可能不准，需要搜索框
- v0.4.2 的「编辑记录」里地点字段也面临同样问题

所以把两个版本共用的底层能力（LocationPicker 组件 + update-record 接口）提前做好，避免重复造轮子。

### 学到的方法

**改「已有数据」比加「新功能」复杂**

原因：新功能出错了重来就行；改已有数据，改完之后整个系统的数据要还是一致的——列表、详情、地图都要对得上。估工作量时要乘以 1.5-2。

**四个规划问题**

规划任何一个功能时，先问自己：
1. 这个功能改的是哪个数据字段？这个字段在哪些地方被用到？
2. 用户有几种路径会用到这个功能？（不只是顺利路径，还有出错、没网、事后补录）
3. 这个功能和已有功能有没有可复用的底层能力？
4. 这个版本做完，下一个版本会不会因为它更容易或更难？

**先想「坏的情况」，再想「好的情况」**

大多数人规划时只跑顺利路径。但真实世界里，顺利路径只占一半，另一半是没权限、没网、数据缺字段、用户中途退出……把「坏的情况」想清楚了，工作量和复杂度才是真实的。

### 对比赛答辩的启发

评委想看的不只是产品 demo，而是：
- 你为什么选这个需求做
- 你规划功能时发现了什么坑、怎么提前绕开的
- 你怎么在有限时间里判断哪些做、哪些不做
- 你踩过什么错，复盘出了什么结论

**业务和代码可以学，但做事的态度和方法要跑在前面。**

## 2026-04-27 · v0.4.1 LocationPicker 实现

### 发生了什么

完成了 v0.4.1：LocationPicker 组件支持 GPS 自动定位 + 逆地理编码 + POI 搜索，详情页「修改定位」功能上线。

### 关键决策

**为什么 LocationPicker 做成底部弹出面板 + 双 tab**

- 「自动定位」解决「我就在这里」的场景
- 「搜索地点」解决「事后补录」的场景，GPS 不准确或用户只想记大概位置时非常有用
- 底部面板不遮挡主内容，手机上操作更自然

**高德 API 调用放在前端还是后端？**

选前端：高德 Key 本来就是通过域名白名单暴露的，官方推荐用法。如果后端 proxy 反而多一层转发，增加延迟和复杂度。

**update-record 接口设计成 partial update**

v0.4.1 更新 location 三字段，v0.4.2 更新 title/species/journal，接口复用。设计时就考虑后续版本，避免重复建接口。

### 踩的坑

**组件化思维不够早**

一开始只在 ListPage 里想怎么改地点，没意识到新建偶遇页以后也需要这个能力。好在 v0.4.1 规划时识别出来了，做成了可复用组件。

### 学到的方法

**边界情况要产品化，不能只是「报错」**

- GPS 权限被拒绝 → 不是弹出 technical error，而是显示「请在系统设置中开启」
- 逆地理编码失败 → 兜底显示「已定位」+ 坐标，不阻塞用户保存
- 搜索结果为空 → 显示「未找到相关地点，请尝试其他关键词」

好的错误提示是产品体验的一部分。

### 技术债记录

- 新建偶遇页的自动定位还没升级成 LocationPicker（用了旧逻辑），等 v0.4.2 完成后统一升级
- POI 搜索目前只返回 5 条，如果搜索结果很多，用户无法翻页（当前用户量下够用）

## 2026-04-27 · v0.5.1 长按多选删除 + 点击跳详情

### 发生了什么

图鉴第二层的"长按多选批量删除"和"点击跳转详情页"看似简单的两个交互，前后改了三轮才落地。中间踩了三个坑，每个都很有教育意义。

### 踩的坑

**坑 1：build 通过 ≠ 功能正常**

第一次实现时，代码里写了 `React.useRef(null)`，但 import 语句只有 `useState, useEffect` —— 根本没引入 React。`npm run build` 居然通过了，但运行时组件直接崩溃，导致**第二层所有事件监听器都没绑定上**——用户点击、长按、右键，全都没反应。

之前我判断"build 成功就算完成"，跳过了浏览器实际验证这一步，错过了这个明显的 ReferenceError。

**坑 2：在 map() 里调用 hooks 是反模式**

```jsx
{records.map((record) => {
  const ref = useRef(null)  // ❌ 违反 Rules of Hooks
  // ...
})}
```

即使 React 正确 import，这种写法每次渲染都会重新创建 ref，导致状态丢失（长按定时器永远不会真正生效）。**正确做法是把照片卡片提取成独立的子组件 `PhotoCard`，hooks 在顶层调用。**

**坑 3：用户说"完全没改变" → 我假设是代码问题，重构了一轮**

代码 + 部署 + 缓存，是三个独立变量。用户报"线上完全没反应"时，我没有先问"你在哪里测的"，而是默认是代码逻辑错了，又重构了一遍。结果发现：**代码改了但根本没 push 到 Vercel**——线上看到的是旧版本。三轮往返多花了大半小时。

### 关键决策

**用 OpenSpec 流程结构化变更**

这次用 `/opsx:propose` + `/opsx:apply` 把改动拆成 proposal → design → spec → tasks 四层，好处是：
- 决策点（如"为什么用定时器区分点击和长按"）被显式记录在 design.md
- 任务列表是可勾选的，避免漏掉边界情况（如 onTouchMove 取消长按）
- 后续归档可以追溯"当时是怎么决策的"

代价是前期慢了一点（5 分钟规划），但避免了边写边改的反复。

### 学到的方法

**Bug 报告排查的第一性原则：先问"在哪"，再问"为什么"**

用户报"不工作"时，按这个顺序追问：
1. **环境**：本地 / 线上 / 移动端？（缓存、部署、分支三大坑都在这一层）
2. **现象**：具体看到什么？（截图、控制台报错）
3. **再现**：什么操作触发的？

之前我经常跳过第 1 步直接进第 2、3 步，假设"代码就是问题"。这次教训深刻。

**「build 通过」不是验收标准，「肉眼看到功能跑起来」才是**

CLAUDE.md 里早就写了"对 UI 或前端改动，启动 dev server 在浏览器实际操作过"。这次跳过了这一步，导致一个明显的 ReferenceError 漏到用户那里。**任何前端改动，编译通过只是起点，浏览器实际跑一遍才算完成。**

**React Hooks 在循环里调用是常见反模式，识别信号：**

看到 `.map(item => { useXxx(...) })` 就要警觉——这几乎一定是错的。正确做法是把循环里的子节点提取成独立组件，让 hooks 在新组件的顶层调用。

### 对比赛答辩的启发

评委如果问"你怎么定位 bug"，可以讲这个故事：
- 第一次失败：build 通过但运行时崩溃 → 教训是"编译 ≠ 运行"
- 第二次失败：诊断方向错了，假设是代码问题，结果是部署问题 → 教训是"先验证环境再验证代码"

这种"踩坑 → 复盘 → 改进诊断流程"的故事，比单纯展示"功能做出来了"有说服力得多。

## 2026-04-27 · v0.5 + v0.5.1 产品视角复盘

### 为什么单独记一份产品视角

之前的复盘几乎都是技术视角（React Hooks、build 通过 ≠ 运行通过、bug 排查方法论）。但回头看本次会话，**真正决定产品质量的不是踩了哪些技术坑，而是做了哪些产品决策**——这些决策当时被夹在"bug 修复"和"功能实现"里，没被识别为产品决策。

这份复盘的目的是把这些决策**显式拎出来**，建立产品直觉。

### 这次会话实际上做了 3 个真正的产品决策（但当时没意识到）

**决策 1：图鉴的"1 张卡片 = 1 个大类"，不是 1 个品种**

- 当时是作为 bug 修复讨论的（"按 species 聚合会把同一只猫拆成多张卡"）
- 真正本质：**用户打开图鉴想看到的是"我遇到过几种动物"，不是"我有多少种猫的细分品种"**
- 按品种聚合：重度用户图鉴变成"橘猫 1 张 + 虎斑猫 1 张 + 英短 1 张……"，成就感被碎片化稀释
- 按大类聚合：「猫（遇到 12 次）」才有"在城市收集动物"的游戏感

**通用规律：** 任何聚合页（图鉴/统计/排行榜），先问两个问题再决定数据维度——
1. 用户视角的 1 个单元代表什么？
2. 重度用户用 1 年后这一页会变成什么样？（这个问题能预测聚合维度选错的灾难）

---

**决策 2：长按多选 = 借用系统范式，不发明新交互**

- kris 原话："类似相册那种长按编辑删除"
- 这句话背后是**借用用户已有的心智模型**（iOS Photos / 微信图片选择都用这个）
- 如果换成"加一个批量管理按钮，进入管理模式"——多了学习成本，体验差一截

**通用规律：** 设计交互前先问"系统/同类产品里这个动作的'母语'是什么"。有现成母语就直接用，发明新交互的门槛要极高。

---

**决策 3：用产品架构兜底 AI 的不稳定性**

- AI 识别同一只橘猫，可能这次说"橘猫"、下次说"虎斑猫，橘色条纹毛"——这是 AI 产品的本质问题
- 解法不是"训练 AI 更准"（不可控），而是**两次 AI 调用做归一化**：
  - 第 1 次：识别具体品种（信息丰富，给详情页用）
  - 第 2 次：映射到稳定大类（聚合稳定，给图鉴用）
- 这是 **AI 产品的通用模式**：用「AI 自由发挥层 + 产品稳定收敛层」两层夹住 AI 的不稳定

**通用规律：** 做 AI 功能时主动用这个夹层模式。不要试图让 AI 完美一致，要在产品层吸收 AI 的偏差。

### 产品价值密度的发现：小改动 ≠ 低价值

| 功能 | 用户感知 | 工程成本 | 价值密度 |
|------|---------|---------|---------|
| v0.5 图鉴第一层网格 | ⭐⭐⭐⭐ | 中 | 高 |
| v0.5 物种大类分类 | ⭐⭐ | 中（多一次 AI 调用） | 中 |
| v0.5.1 长按多选删除 | ⭐⭐ | 中 | 中 |
| **v0.5.1 点击跳转详情** | **⭐⭐⭐⭐** | **低（半小时）** | **极高** |

**洞察：点击跳转详情这件事工程成本最低、用户感知价值最高**——它把图鉴从"展示"变成"可探索"，补全了"图鉴 ↔ 时间线 ↔ 地图"的链接。但因为它"看起来太简单"，最初没作为独立功能规划，被埋在 v0.5.1 里和长按多选一起做。

**通用规律：** 不要被"小改动"骗了。**很多极高价值的功能恰恰是改动小的——因为它们补全了产品的链接，而不是新增内容**。规划版本时专门列一栏"打通现有功能的小改动"，优先级高于"加新功能"。

### 暴露的产品流程问题

**问题 1：验收没产品化**

每次反馈用的是用户语言（"图鉴里只有一种叫其他"、"完全没改变"、"长按没反应"）。这些反馈很珍贵，但处理是 ad-hoc 的，没沉淀成结构化 checklist。

**改进：** 给图鉴这种核心页面建一个 **30 秒可执行的冒烟 checklist**，每次部署后自己跑一遍。这 5 个 yes 比"代码改完了"更重要：
1. 图鉴第一层有几个大类？数据对不对？
2. 点击大类能进第二层吗？
3. 第二层照片能点开详情吗？
4. 长按/右键能进选择模式吗？
5. 删除一张照片能正确减少计数吗？

---

**问题 2：版本节奏被"急于推进"绑架**

时间线：19:04 v0.5 基础 → 19:46 直接验收（没充分自测）→ 验收漏掉归"其他"问题 → 返工 → 20:19 v0.5.1 又出 React 崩溃 + 没 push → 加倍返工。

模式：**急于推进版本号 → 验收草率 → 用户测试时才发现问题 → 加倍时间返工。**

**改进：** 每个版本上线后强制 **5 分钟"用户体验时间"**，期间不写代码、不和 AI 对话，纯粹当用户用一遍。这 5 分钟省下的返工时间是 30 分钟起步。

---

**问题 3：和 AI 协作时反馈语言之间需要"翻译层"**

"长按没反应"作为反馈，AI 默认假设是代码问题——但真相可能是部署/缓存/分支。三轮往返耗时 45 分钟。

**改进：** 反馈"功能 X 不工作"时，主动加三句话——
- 在哪里测的（线上 / 本地 / 手机）
- 看到了什么（截图、控制台报错）
- 做了什么操作

这三句话能让 AI 排查效率翻倍。

## 2026-04-30 · v0.6 偶遇报告 + v0.7 分享功能

### 1. Web 分享能力的现实边界与降级策略

**背景**  
这次 v0.7 的分享功能，用户直觉上想要的是“点一下就发到微信/小红书等常用 app”。但在 Web 环境里，这类能力并不稳定：不同浏览器对 `navigator.share`、剪贴板、下载行为的支持差异很大，同一个功能在 Chrome 可用，在夸克或内置浏览器里可能就是假支持或完全失效。

**决策**  
最终没有把“直接跳常用 app”作为核心能力，而是把分享拆成三层：
- 主路径：下载图片
- 备用路径：复制到剪贴板 / 复制文案
- 增强能力：支持时显示 `分享到...`

同时对已知假支持环境做降级处理，不让用户看到一个点了没反应的按钮。

**为什么这样做**  
用户真正要完成的是“把内容发出去”，不是“必须通过某个浏览器能力发出去”。与其追求理想化但不稳定的直跳体验，不如优先保障最稳定的可完成路径。  
这也是一个典型的产品取舍：先保成功率，再保便捷性。

**被否定的方案**  
- 给常用 app 做 deep link 直跳  
  原因：兼容性差，浏览器拦截不稳定，维护成本高。
- 把 `分享到...` 当成主按钮  
  原因：浏览器支持差异太大，容易误导用户。
- 所有浏览器一视同仁显示系统分享  
  原因：会出现“按钮存在但无效”的坏体验。

**下次怎么做**  
以后做依赖浏览器或系统能力的功能时，先画清楚三层结构：
- 哪条是最稳定主路径
- 哪条是降级路径
- 哪条是增强路径  
不要一开始就把“最酷的能力”当成核心路径。

### 2. 报告页的数据成熟度分层设计

**背景**  
v0.6 的报告页一开始遇到一个问题：记录少的时候，到底该不该显示完整报告？  
如果完全不显示，用户会觉得功能“没内容”；如果 1 条数据也硬做完整报告，又会显得很傻、很空、没有说服力。

**决策**  
最后采用了按数据成熟度分层展示的方式：
- 0 条：空状态
- 1-2 条：只显示轻提示，不展示完整报告
- 3-4 条：展示完整报告，同时提示“数据还在积累中”
- 5 条及以上：正常完整报告

同时对不同时间范围做信息裁剪：
- 最近三个月：不显示“最活跃月份”
- 今年：显示“最活跃月份”

**为什么这样做**  
报告类功能不是单纯判断“有没有数据”，而是要判断“数据是否足以支撑这个表达”。  
完整报告是一种较强的内容表达，如果样本太少，硬展示会削弱可信度和仪式感；但如果用户已经有 3-4 条记录，还完全不给看，又会让体验过于保守。

**被否定的方案**  
- 少于 5 条一律不显示完整报告  
  原因：太粗暴，挡掉了用户已经拥有的信息。
- 只要有 1 条就显示完整报告  
  原因：表达过度，内容质量不成立。
- 最近三个月和今年使用完全同一套报告结构  
  原因：时间窗口不同，信息价值不同，“活跃月份”在三个月视角里信息密度偏低。

**下次怎么做**  
以后做任何“总结型/成就型/统计型”页面，不只问“有没有数据”，还要先定义：
- 数据少时怎么表达
- 数据刚够时怎么表达
- 数据丰富时怎么表达  
内容型功能应该按成熟度开放，不适合一刀切。

### 3. 分享卡片从“整齐”到“可读”的 UI 取舍

**背景**  
分享卡片的设计经历了几轮调整：
- 第一版：信息全塞进去，容易乱
- 第二版：布局整洁了，但日志太短、留白过多，内容表达不够
- 第三版：改成内容优先，缩小图片占比、扩大日志空间、压缩边缘空白

**决策**  
最后确认一个核心原则：分享卡片不是页面截图，也不是纯视觉海报，而是“传播场景里的内容卡”。  
因此优先级调整为：
- 第一优先：用户一眼看懂这是一次什么偶遇
- 第二优先：日志有足够内容可读
- 第三优先：视觉整洁和风格统一

**为什么这样做**  
用户分享这张卡，不只是分享一张动物照片，而是在分享“一次偶遇记录”。  
如果日志被压得太短，虽然画面整齐，但分享价值会下降；如果信息过多又会显得拥挤。  
所以设计重点不是“放多少元素”，而是“让什么信息被优先看到”。

**被否定的方案**  
- 把页面里的所有信息照搬到卡片上  
  原因：信息层级混乱，卡片会乱。
- 为了整洁极度压缩内容  
  原因：内容表达不够，分享动机变弱。
- 把它做成单纯摄影海报  
  原因：不符合产品“记录偶遇”的核心价值。

**下次怎么做**  
以后做分享物料时，先回答这三个问题：
- 用户真正想分享的核心是什么
- 对方扫一眼最先该看到什么
- 哪些信息可以删，哪些信息必须保  
分享设计的本质是内容重构，不是界面缩略。

## 方法论提炼

### 4. 先做主路径，再做增强能力

这次分享功能最重要的经验，不是“怎么把系统分享做出来”，而是明确了能力优先级。  
做产品时，应该先找出最稳定、最可完成的主路径，再去叠增强能力。否则很容易把不稳定特性误当核心功能，最后用户反而完不成任务。

**可复用原则**
- 主路径优先保证成功率
- 增强能力只在支持环境里加分
- 不让降级场景承担主流程

### 5. 内容产品要按数据成熟度分层开放

报告页这次说明了一点：很多内容功能不是“有/无”问题，而是“成熟/不成熟”问题。  
产品不该只判断功能做没做完，还要判断当前数据是否足以支持这个内容表达。

**可复用原则**
- 空数据给空状态
- 弱数据给轻反馈
- 半成熟数据给简版结果
- 成熟数据给完整表达

这个方法以后也可以用在成就系统、图鉴完成度、回头客识别等功能上。

### 6. 浏览器差异是产品约束，不是偶发 bug

这次 Chrome 和夸克的差异说明，Web 环境能力不一致不是例外，而是常态。  
所以产品设计不能假设所有浏览器表现一致，而要从一开始就准备：
- 能力检测
- 降级方案
- 环境差异下的主次路径切换

**可复用原则**
- 浏览器兼容不是上线后补救
- 应该前置到产品方案里
- “在某些环境不可用”不一定是 bug，可能是边界条件

### 7. 产品方向要先对齐，再追求执行速度

这轮开发里，一个非常关键的教训是：技术实现可以很快，但产品方向不能默认由执行者顺手决定。  
尤其是涉及交互流程、文案语气、信息架构、展示优先级、分享路径、数据阈值这类问题时，真正重要的不是“能不能马上做出来”，而是“是不是应该这样做”。

这次过程中其实已经出现过几类典型情况：
- 功能可以很快补出来，但产品表达是否合理还没对齐
- 一个交互 technically 可行，但不一定符合用户预期
- 一个 UI 改动看上去只是样式调整，实际是在替产品做信息优先级判断
- 一个降级方案写起来很顺手，但到底该隐藏、提示还是保留按钮，属于产品决策，不是纯实现细节

这说明一个常见误区：  
**执行速度快，很容易让人误以为决策也可以一起快。**  
但产品工作恰恰相反，越是实现门槛低、改动成本小的地方，越容易在不知不觉中把“实现判断”替换成“产品判断”。

所以后续协作里，需要明确区分两类事情：

**可以直接快做的事情**
- 明确 bug 修复
- 已经拍板方案下的实现细节
- 明确目标下的兼容性修正
- 不改变产品语义的技术性调整

**必须先讨论再动手的事情**
- 交互流程变化
- 默认行为变化
- UI 方向变化
- 信息展示层级变化
- 文案语气变化
- 降级策略选择
- 数据展示规则和阈值判断

这里真正要保护的，不是“慢一点更稳”，而是：  
**产品判断权不能在高执行速度下被无声吞掉。**

对个人做产品来说，这条尤其重要。因为独立开发或高频协作时，最容易滑向一个状态：
- 讨论还没结束
- 代码已经写完
- 方案看起来也“能用”
- 结果后面又要花更多轮去修正方向

短期看像是快，长期看其实是反复返工。

所以更好的工作方法是：

1. 先判断这次改动属于“执行问题”还是“方向问题”
2. 如果是方向问题，先把原则和边界说清楚
3. 一旦方向达成一致，再快速实现
4. 实现阶段尽量不要再次偷偷替产品做默认决策

这条原则的价值不只在减少误改，更重要的是它会提升协作效率：
- 讨论时只讨论方向，不陷入代码细节
- 实现时只追求落地速度，不反复摇摆
- 复盘时也能清楚区分：到底是判断错了，还是执行没跟上

**可复用原则**
- 快速实现不等于快速决策
- 产品方向先对齐，执行再提速
- 越是影响用户感知的改动，越不能默认替用户拍板
- 把“哪些事必须先讨论”写进协作文档，能显著减少后续返工

### v0.6 之前必须想清楚的产品策略问题

**问题 1：产品定位是「工具」还是「游戏」？**

当前的图鉴 + emoji + 像素风强烈指向"游戏化收集"，但核心交互（拍照 + AI 识别 + 保存）又像"日记工具"。两种定位的演进路径完全不同：
- **工具路径**：v0.6 做导出、统计、搜索、跨设备同步
- **游戏路径**：v0.6 做成就系统、打卡日历、稀有度系统、好友炫耀

这个答案决定 v0.6/v0.7/v1.0 的所有功能选择。

---

**问题 2：「记录」和「分享」哪个是核心？**

- 时间线 + 图鉴 + 地图 = 私人查看
- 但分享按钮一直是"敬请期待"

如果分享是核心，整个产品应围绕"生成一张可炫耀的偶遇卡"设计——AI 文案要有梗、图鉴成就感要强、分享链接要好看。

如果分享只是辅助，明确告诉自己"这是私人记录工具"，停止把"分享"当占位。

---

**问题 3：用户量增长前最先崩的是什么？**

技术债清单里记的是"图片 base64 存数据库"。但产品视角看更早崩的可能是：
- **新用户引导路径**：打开 app 是空白页，不知道第一步该干啥
- **AI 识别失败的产品体验**：现在是错误提示，应该是"AI 失败也能保存（手填即可）"——其实已做但没被强调
- **国内访问困难**：vercel.app 域名 = 没有真正可触达用户

这三个里，技术含量最低的（域名）反而对产品可触达性影响最大——它需要钱和决心，不需要写代码。

### 学到的方法

1. **聚合页两问法**：1 个单元代表什么？重度用户 1 年后这页变成什么样？
2. **交互"母语"原则**：先找系统/同类产品的现成范式，再考虑发明新交互
3. **AI 夹层模式**：自由发挥层 + 稳定收敛层，吸收 AI 的偏差
4. **小改动可能极高价值**：版本规划专列"打通现有功能的小改动"
5. **5 分钟用户体验时间**：上线后强制冷却，纯当用户用一遍
6. **反馈三句话**：在哪 / 看到啥 / 做了啥，让 AI 排查效率翻倍

### 对比赛答辩的启发

可以讲一个"被埋没的产品决策"的故事——
- 表面：v0.5 上线后发现物种全归"其他"，改了 4 次 prompt 才解决
- 真相：这是个**聚合维度选错**的产品判断问题，不是 AI 问题
- 反思：作为产品经理，**我开始有意识地把"产品决策"和"技术问题"区分开**——之前所有问题在我脑子里都是混在一起的"bug"

这种"从技术视角升级到产品视角"的故事，比展示具体功能更有说服力——评委想看的是**产品经理的成长**，不是产品本身。

## 2026-05-01 · 从0-1到1-100的认知转变

### 发生了什么

v0.7完成后，在规划v0.8-v1.0的过程中，和AI讨论了"PM的工作不只是做功能"这个问题，触发了几个认知转变。

### 关键认知

**做功能本质上是在提出假设，但我一直没有验证它**

v0.1到v0.7，每次做新功能的逻辑都是"我觉得用户需要这个"，然后就做了。做成就系统和做地图的决策方式完全一样，没有问过"用户真的会用吗"，也没有想好"用什么数据证明这个判断是对的"。这是执行者和决策者的根本区别。

0-1阶段这样做没问题，快速出东西比过度分析重要。但如果不意识到这个转变，1-100阶段继续这样做就会导致方向漂移——做了一堆用户不需要的功能。

**AI产品有普通产品没有的一层复杂度**

普通功能出bug，改代码就行。AI功能"出bug"，可能是模型问题，改不了。所以AI产品PM需要额外想：AI失败了产品怎么办？AI给出不一致的结果产品怎么办？

回头看这个项目，其实有两个好决策，但当时都是被逼出来的，不是主动设计的：
- AI识别失败后用户还能手填保存（产品层兜住了AI的失败）
- 两次AI调用做物种归一化（产品层收敛了AI的不一致）

这两个决策下次应该是主动想到的，而不是遇到问题了才想到。

**数据是用来解释决策路径的，不只是展示结果**

"我做了功能A，效果不错"是一层。"我考虑了A/B/C三个方案，基于XX选了A，数据说明这个选择是对的"是另一层。

growth.md里一直在写"有哪些选项被否定了"，但之前都是凭直觉说被否定了，没有数据支撑。有了benchmark或埋点数据，"为什么不是B"就变得有说服力。这是从"凭感觉做决策"升级到"有数据解释决策路径"。

### 下次怎么做

做每个功能之前，先写一句话：我在假设什么，用什么验证。

做AI功能之前，先问：AI失败了会怎样？AI给出不一致的结果会怎样？

做完功能之后，不只记录"做了什么"，还要记录"考虑了哪些选项、为什么选这个"——如果有数据支撑更好。

## 2026-05-01 · 技术实现性优化复盘

### 发生了什么

围绕比赛评分里的“技术实现性”维度，对项目做了一轮工程向优化，包括 API 错误处理和日志收口、核心纯逻辑补最小单测，以及首屏拆包和按需加载。

### 学到的方法

这轮技术优化确认了一点：项目接近验收阶段时，不应优先做高侵入式重构，而应优先选择低风险、高收益、易验证的优化动作。

以性能问题为例，首屏包体过大时，先通过页面懒加载和 vendor 拆包延后非首屏资源加载，比直接调整整体架构更稳妥。性能优化应先解决最确定、最容易验证的问题，再决定是否继续深入。

Demo 的稳定性更多取决于异常路径的设计，而不只是成功路径是否通畅。接口失败、地图加载失败、定位不可用、配置缺失等情况，如果只是静默失败或表现为“没有反应”，会明显削弱项目的技术完成度。

即使出现异常，只要系统能够给出明确提示、保留可恢复路径，并让开发者能够快速定位原因，整体观感仍然是可控和专业的。

这轮也再次确认了一个基础但容易混淆的工程认知：`commit`、`push` 与 `deploy` 是三条独立链路。`commit` 负责记录本地版本历史，`push` 负责同步到远程仓库，`deploy` 负责发布到运行环境。

三者分别对应版本记录、代码协作和线上交付，不能互相替代。后续做验收、发布和版本确认时，需要始终将这三件事分开检查。

## 2026-05-01 · PM 成本/体验权衡决策框架

### 发生了什么

规划 v0.8「回头客识别」功能时，遇到两个核心设计问题：
1. **识别策略选择**：用相似度打分vs. 二元判断 vs. 平衡方案？
2. **API 成本权衡**：每次保存都调 AI（高成本但体验好）vs. 用户主动查找（低成本但容易被忽视）vs. 后台周期检测（成本恒定但不实时）？

这两个问题没有"标准答案"，需要根据**当前产品阶段**和**优先级判断**来决策。

### 关键决策

**决策 1：识别方案采用「相似度打分」（0-100 分）**

方案比较：
| 方案 | 优点 | 缺点 |
|-----|-----|-----|
| 二元判断（是/否） | 简单清晰 | 失去准确度信息，容易被当成确定判断 |
| 相似度打分（0-100） | 透明、可解释、给用户自主判断的空间 | 实现稍复杂，需要 AI 返回分数 |
| 分级（低/中/高） | 折中方案 | 精度介于两者之间，信息丰富度也介于之间 |

**最后选择打分的理由**：
- 用户能看到"和 3 月 15 日那只橘猫的相似度 87%"，比"是/否"有信息量
- 如果以后要调整阈值，打分方案有调优空间，二元方案就废了

---

**决策 2：API 调用策略选「每次保存自动调」**

核心权衡：**当前阶段优先级 > 成本优化**

| 方案 | 成本 | 用户体验 | 何时用 |
|-----|-----|---------|--------|
| 每次保存调 | 高（每条记录都调） | ⭐⭐⭐（即时反馈） | **v0.8-v0.9 验证期**（现在） |
| 用户主动查找 | 低 | ⭐⭐（需主动操作） | 数据量 1000+ 时的优化方案 |
| 后台周期检测 | 恒定 | ⭐（延迟反馈） | 大规模运营期的方案 |

**为什么选「每次保存调」**：
1. **数据量判断**：当前用户记录有限，同物种最近 10 条的成本不会爆炸
2. **优先级判断**：v0.8 是功能验证期，用户体验 > 成本优化
3. **监控点**：架构上留口子，当 API 调用 2-3 倍后自动告警，再做成本优化
4. **用户行为**：实时反馈能建立用户预期（保存 → 立即提示），更容易形成习惯

---

### 学到的方法

**「成本 vs 体验」权衡的 4 个问题框架**

每次遇到这类权衡决策时，依次问自己：

1. **当前产品阶段是什么？**
   - MVP 验证期？重点是用户体验和功能验证
   - 增长期？重点是规模化和可靠性
   - 成熟期？重点是效率优化和成本控制
   
   **→ 不同阶段的优先级不同。v0.8 还在 MVP 验证期，体验优先**

2. **成本的「爆炸点」在哪？**
   - 有明确的数据量上限吗？（比如"同物种最近 10 条"）
   - 还是成本会随着用户增长线性攀升？
   - 提前规划成本告警线
   
   **→ 如果有数据量上限，在触碰之前不用急着优化**

3. **体验差异有多大？**
   - 方案 A 和方案 B 对用户感受的差异大吗？
   - 是"看不出区别"还是"明显更好"或"明显更差"？
   
   **→ 差异越大，越应该优先满足体验好的方案**

4. **能否为未来优化留口子？**
   - 是否能在架构上支持"从方案 A 升级到方案 B"而不大改？
   - 成本优化是否可以是后续迭代，不影响当前版本？
   
   **→ 如果能留口子，就优先上用户体验更好的方案**

---

### 被否定的方案与理由

**为什么不是「分级判断」（低/中/高 相似度）**
- 实现简单，但信息密度低
- 用户看"中"和"高"的区别不大，还不如直接看打分

**为什么不是「后台周期检测」（经济高效）**
- 成本虽然最低，但用户下周才能看到提示，反馈延迟大
- 不符合"新功能需要强化用户预期"的阶段需求

**为什么不是「用户主动查找」（保守方案）**
- 成本确实低，但"用户需要主动点按钮才能看到"这个体验不够好
- 容易被用户忽视，失去功能价值

---

### 对比赛答辩的启发

如果评委问"你怎么做产品决策"，可以用这个例子：

> 在 v0.8 规划时，遇到了"API 成本 vs 用户体验"的权衡。我没有选"成本最低"的方案，而是选了"用户体验最好"的方案，因为：
>
> 第一，我判断了产品阶段——v0.8 还在 MVP 验证期，重点是让用户快速体验到功能价值，而不是优化成本。
>
> 第二，我评估了成本的上限——因为有"同物种最近 10 条"的过滤，即使是每条都调 API，成本也不会失控。
>
> 第三，我为未来留了调整口子——如果数据量真的大了，可以改成后台周期检测，不需要大改架构。
>
> 这说明我的决策不只是"选体验好的"，而是综合考量了当前约束、数据量上限、未来升级空间——最后做出的选择是**有根据的**。

---

### 可复用的决策框架

以后遇到类似的权衡问题，用这个框架自问：

```
1. 当前阶段优先级是什么？（体验 > 成本 还是 成本 > 体验）
2. 成本爆炸点在哪？（是否有可控的数据量上限）
3. 用户感受差异多大？（明显差异 vs 难以察觉）
4. 能否为升级留口子？（是否支持未来从 A 方案升级到 B 方案）

答完这四个，再选方案。
```

这套框架可以应用到：
- API 成本决策（本次）
- 功能完整度 vs 上线速度（v0.1-v0.7 的每次决策）
- 前端兼容性 vs 功能丰富度（v0.7 分享卡片的决策）

## 2026-05-01 · v0.8「回头客识别」实现策略与代码复用

### 发生了什么

v0.8 的核心需求是"识别同一只动物的重复出现"。这个功能涉及数据库设计、AI 对比逻辑、前端弹窗、用户确认四个环节，是目前最复杂的一个版本。

最有价值的决策是：**提取共用的检测引擎，而不是在两个地方（保存后 + 手动查找）分别复制代码。**

### 关键决策

**决策 1：创建 `_lib/detectReturning.js` 统一检测逻辑**

起初考虑直接在 `api/save-record.js` 里内联检测代码（快速上线），但想到：
- v0.8 需要两个入口调用检测：保存时自动 + 详情页手动
- 如果直接复制代码，后续修改检测逻辑时要改两个地方
- 同步两个地方的 bug 修复成本高

**最终决策**：提取独立的 `performDetectReturning` 函数，同时在 `detect-returning.js` 和 `save-record.js` 中复用。

**成本**：提取函数多花 10 分钟
**收益**：避免代码重复 + 未来修改检测规则只需改一个地方

---

**决策 2：检测结果返回「完整对象」而非「仅 ID」**

一开始想返回 `{ detected: true, similarRecordId, score }`，但弹窗需要展示关联记录的"日期、物种、标题"这些信息。

两个方案：
1. 前端拿到 `similarRecordId` 后再去 `records` 列表里查找对象
2. 后端直接返回 `{ detected: true, similarRecord: {...}, score }`

**最终选择方案 2**，因为：
- 后端已经查询过了，多返回一个对象成本低
- 前端不需要额外的"根据 ID 查找"逻辑
- 减少前端依赖于列表状态的耦合

---

**决策 3：前端状态管理「分离关注点」**

在 `NewEncounterPage` 中处理保存后的弹窗时，分离了三个状态：
- `suggestedRecord`（弹窗显示的关联记录）
- `suggestedScore`（相似度分数）
- `saveRecord`（刚保存的新记录，用于确认时更新）

虽然可以合并成一个 `suggestion` 对象，但分离后更清晰：每个状态对应一个 UI 需求，改一个需求时不用动其他状态。

---

### 学到的方法

**「共用逻辑」的识别信号**

如果一个逻辑会被调用 2+ 次（不管是立即调用还是将来可能调用），就应该提取出来：
1. 减少维护成本
2. 提高单一职责
3. 便于单测

这个决策的成本是"多花一点点时间做抽象"，收益是"未来所有修改都能一次到位"。

---

**API 返回结果「返回足够的信息」**

后端检测完成后，如果只返回 ID，前端还需要额外工作找对象。这不是"前端自己想办法"的问题，而是**后端应该认识到前端会需要什么**。

一个好的 API 设计，不只是"有没有正确字段"，还要考虑调用方会做什么，提供足够的信息让调用方的代码更简洁。

---

**前端状态细粒度设计**

`NewEncounterPage` 有很多状态（`isLoading`, `error`, `selectedImage`, `speciesTag` 等等），这次新增 3 个弹窗相关的状态。

如果直接把它们塞进一个 `modal` 对象，反而会增加理解成本。分离状态虽然多写了 3 行 `useState`，但每个状态的含义更清晰。

### 被否定的方案与理由

**为什么不在每个 API 里内联检测逻辑**
- 代码重复，维护成本高
- 两个地方如果逻辑不一致，调试困难

**为什么不让前端根据 ID 查找关联记录**
- 增加前端复杂度，需要依赖 `records` 列表已经加载
- API 已经有这个数据了，重复查询浪费

**为什么不合并弹窗相关的三个状态**
- 虽然可以用对象，但拆分开更容易理解
- 改其中一个参数时，不会意外影响其他参数

---

### 可复用的决策框架

以后实现相似的功能时，可以参考这个决策路径：

```
1. 这个逻辑会被多个地方调用吗？
   → Yes: 提取函数
   → No: 可以内联

2. 后端已经有这个数据了吗？
   → Yes: 返回给前端，不让前端重复查询
   → No: 由前端去获取

3. 前端状态是否可以合并？
   → 如果合并会让含义不清晰，就拆开
   → 如果合并能显著减少 useState 数量，考虑合并
```

## 2026-05-01 · v0.8 物种颗粒度选择：大类 vs 中粒度 vs 品种级

### 发生了什么

回头客识别需要定义"什么叫同一只动物"。粒度选择决定搜索范围、识别准确性、API 成本。

### 关键决策

**最终选「中粒度」（橘猫、泰迪、乌鸦），而不是大类（猫、狗）或品种级**

- **大类太粗**：同物种 100 条记录，同一只个体无法区分
- **中粒度最优**：同物种最近 10 条，大概率是同一只，且 AI 识别成功率高（80%+）
- **品种级太细**：无法穷举，AI 每次描述都不同（无法稳定 mapping）

同时建立了穷举 + 兜底方案：第二层 AI 调用必须 map 到预定义列表，不在列表的归"其他"。这是产品层对 AI 不稳定性的兜底。

### 学到的方法

**数据聚合粒度的选择标准**：既能让搜索范围稳定在 N=5-10 条，AI 识别失败率可控（70%+ 准确），又能用穷举方案收敛的粒度。

这个思路可以复用到：消息聚合粒度、统计维度选择、推荐系统的特征选择。
